# Scenario 01 — Batch Incremental Ingestion

**Phase:** 1 – Ingest  
**Duration:** 45 min  
**Difficulty:** Intermediate  
**Domain:** Generic (Orders)

---

## Learning Objectives

1. Build a watermark-based incremental pipeline in Data Factory that reads only new/changed rows.
2. Write raw records append-only to a Bronze Delta table with a partitioned ingest date.
3. Store and update watermarks atomically so re-runs are safe and idempotent.

## Prerequisites

- Shared baseline complete (see [baseline/](../../baseline/))
- `lh_advanced_scenarios` Lakehouse exists
- Control tables (`bronze.pipeline_watermark`, `bronze.pipeline_errors`) created via the notebook in this scenario

## Architecture

```
HTTP source (synthetic CSV)
       │
       ▼
Data Factory Pipeline: pl_bronze_orders_ingest
  ├─ Lookup: read last watermark from bronze.pipeline_watermark
  ├─ Copy Activity: GET /orders?since=<watermark> → Files/landing/orders/
  ├─ Notebook Activity: nb_01_bronze_orders_load (Bronze Delta write)
  └─ Stored Procedure Activity: update watermark
```

## Dataset

Synthetic orders CSV generated by [generate_orders.py](../shared/generate_orders.py).

| Column | Type | Notes |
|---|---|---|
| order_id | STRING | Business key |
| customer_id | STRING | FK |
| order_date | DATE | |
| updated_at | TIMESTAMP | Watermark column |
| status | STRING | NEW / SHIPPED / CANCELLED |
| amount | DECIMAL(10,2) | |

Seed file: `Files/landing/orders/seed/orders_seed.csv` (~10,000 rows).  
Incremental files drop in `Files/landing/orders/<YYYY-MM-DD>/`.

## Step-by-Step Instructions

### Step 1 — Bootstrap the control tables

Open notebook **`01_bootstrap_control_tables`** and run all cells.  
It creates `bronze.pipeline_watermark` and `bronze.pipeline_errors` if absent.

### Step 2 — Load the seed data as Day 0

Run notebook **`01_bronze_orders_seed`**. It reads the seed CSV, adds `_ingest_date`
and `_source_file` audit columns, and writes:

```python
df.write \
    .format("delta") \
    .mode("overwrite") \
    .partitionBy("_ingest_date") \
    .option("overwriteSchema", "true") \
    .saveAsTable("lh_advanced_scenarios.bronze.orders_raw")
```

After writing it sets the initial watermark to the max `updated_at` in the seed.

### Step 3 — Import and publish the pipeline

Import [pipeline/pl_bronze_orders_ingest.json](pipeline/pl_bronze_orders_ingest.json)
into the Fabric workspace.

Key pipeline parameters:

| Parameter | Default | Description |
|---|---|---|
| `p_entity` | `orders` | Entity name (used in watermark lookup) |
| `p_source_url` | _(set in linked service)_ | Source REST endpoint |
| `p_landing_path` | `Files/landing/orders` | Drop folder in Lakehouse Files |

### Step 4 — Run the pipeline incrementally

Trigger the pipeline manually with today's date.  
The Lookup activity fetches the last watermark, the Copy Activity downloads only
rows where `updated_at > watermark`, and the Notebook Activity appends to Bronze.

### Step 5 — Verify idempotency

Re-trigger the pipeline. Confirm:
- No new rows appended (source has no new changes)
- Watermark value unchanged
- Bronze Delta history shows one additional `WRITE` with 0 added files (no-op path)

## Expected Outputs

| Artifact | Location | Check |
|---|---|---|
| Bronze table | `lh_advanced_scenarios.bronze.orders_raw` | `SELECT COUNT(*) FROM bronze.orders_raw` → 10,000 |
| Watermark row | `bronze.pipeline_watermark` | `WHERE entity_name = 'orders'` → 1 row |
| Delta partition | `bronze.orders_raw/_ingest_date=<today>` | Partition directory visible in Lakehouse explorer |

## Idempotency Check

```sql
-- Run twice; expect identical results
SELECT _ingest_date, COUNT(*) AS cnt
FROM   bronze.orders_raw
GROUP  BY _ingest_date
ORDER  BY _ingest_date;
```

## Troubleshooting

| Symptom | Likely Cause | Fix |
|---|---|---|
| Lookup activity fails | Watermark table not created | Run bootstrap notebook first |
| Duplicate rows after re-run | Watermark not updated | Check SP activity in pipeline; verify `updated_at` column type is TIMESTAMP |
| Schema mismatch on append | Source added columns | Set `mergeSchema = true` in Bronze write; log drift alert |

## Acceptance Criteria

- [ ] `bronze.orders_raw` contains seed rows after Day 0 load
- [ ] Incremental run adds only new rows (verified by partition counts)
- [ ] Second incremental run with no source changes produces no new rows
- [ ] Watermark table has exactly one row per entity with an updated timestamp
